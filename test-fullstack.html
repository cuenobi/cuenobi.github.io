FLOWFUSION TECHNOLOGY – Full-Stack Code Challenge

This page outlines the full‑stack coding assignment for candidates applying for the role of Full‑Stack Engineer at FlowFusion Technology.

The challenge requires you to build both a back‑end API service and a front‑end web application, following engineering best practices and ensuring proper end‑to‑end integration between both systems.

Goal: Build a full authentication system consisting of:
  • A back‑end authentication API (Golang + Fiber + PostgreSQL)
  • A front‑end web interface (React or Next.js)
  • Both must integrate seamlessly via API communication

1. Problem Overview

You must implement a fully functional authentication flow where the front‑end interacts directly with your back‑end API. The system should include login, registration, protected routes, and logout.

1.1 Features Required

  • User registration
  • User login
  • JWT Access Token + Refresh Token flow
  • Protected user info endpoint (/me)
  • Front‑end UI pages (login, register, dashboard)
  • Logout behavior on front‑end
  • Front‑end must call your implemented back‑end API directly

2. Functional Requirements

2.1 Back-End Requirements

The back-end must be implemented using:

  • Language: Go (Golang)
  • Framework: Fiber
  • Database: PostgreSQL with GORM
  • Authentication: JWT (access + refresh token)
  • Architecture: Clean Architecture

Required API Endpoints:

  1. POST /auth/register – register a new user
  2. POST /auth/login – login and receive tokens
  3. POST /auth/refresh – refresh tokens
  4. POST /auth/logout – clear tokens / invalidate session
  5. GET /me – get current user info (protected)

2.2 Front-End Requirements

The front-end must be built using:

  • React (Vite or CRA) or Next.js

Minimum UI Requirements:

  • Login page
  • Register page
  • Dashboard / profile page (protected)
  • Logout feature
  • Client-side state for authentication (localStorage, sessionStorage, or cookies)

The UI must call your back‑end API for all authentication actions. No mock API is allowed for this challenge.

3. Technical Requirements

3.1 Integration Requirements

The front‑end must successfully connect to the back‑end API. At minimum:

  • Front‑end /login → calls POST /auth/login
  • Front‑end /register → calls POST /auth/register
  • Front‑end protected page → calls GET /me
  • Front‑end logout → clears tokens + optionally calls /auth/logout

CORS must be configured correctly so the front‑end can communicate with the API from a different port.

3.2 Back-End Architecture

  • Use clean separation of layers: Domain, Use‑Case, Interface, Infrastructure
  • No framework‑specific types or logic inside domain or use‑case layers
  • Use environment variables for DB + JWT secrets

3.3 Front-End Architecture

  • Use reusable components
  • Folder structure should be clean and scalable
  • Validation for input fields
  • Handle error messages (invalid login, existing email, etc.)

3.4 Bonus Options (Optional)

  • Redis for token/session management
  • Docker Compose to start both front‑end and back‑end
  • Cypress E2E tests

4. Non-Functional Requirements

We will evaluate based on:

  • Code readability
  • Simplicity
  • Security awareness
  • Performance considerations
  • Front-end UX quality

5. Project Setup & Deliverables

Your repository must include:

  1. Back‑end source code (clean architecture)
  2. Front‑end source code (React or Next.js)
  3. Both must run and connect together
  4. A complete README.md including:
    • Setup instructions for back‑end
    • Setup instructions for front‑end
    • Environment variables required
    • API usage examples
    • How the front‑end communicates with the back‑end
  5. Optional: Docker Compose file

6. Bonus: Tests

Optional test additions:

  • Back‑end unit tests
  • Front‑end E2E tests via Cypress

7. Submission Guidelines

  • Provide your GitHub/GitLab repository link
  • Ensure both apps run without errors
  • Ensure back‑end API is reachable by the front‑end
